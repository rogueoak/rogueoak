# 0001 - rogueoak.com one-page site

## Problem

rogueoak.com has no home. Rogue Oak ships three products - Spectra, Trellis, and Canopy - but
there is nowhere on the web that tells the story of the name and shows the work off in one place.
This spec builds the first version of that site: a single page that a visitor lands on, reads the
Rogue Oak story, and sees the three projects. It is the foundation the later infra specs
(CI, container, release, deploy) build on, so it also establishes the app skeleton and the Canopy
wiring the whole site draws from.

Audience: developers and prospective users evaluating Rogue Oak's tools, plus anyone curious about
the name.

## Outcome

- `npm run dev` serves a single page at `/` that renders, in order:
  1. A **hero** with the Rogue Oak wordmark/avatar and the master tagline
     "Quietly rogue. Seriously solid."
  2. The **name story** - a short section on the lone oak: strength, longevity (300 years growing,
     300 living, 300 dying), standing apart from the grove, endless value to its ecosystem, tied
     back to "Rogue Oak aims to build incredible customer value."
  3. A **projects showcase** - three cards (Spectra, Trellis, Canopy), each with the product logo,
     name, one-line pitch, 2-3 benefits, and a link to its repo/site.
  4. A minimal **footer** (Rogue Oak, GitHub org link, "built with Canopy").
- The page is fully themed by Canopy tokens and works in light and dark, toggled by a control that
  flips the `.dark` class; the theme choice persists and does not flash on load.
- Sections and images **fade/rise into view as the visitor scrolls** - each reveals once when it
  enters the viewport. The effect is subtle and respects `prefers-reduced-motion` (no animation for
  visitors who ask for less motion); content is never hidden without JS (progressive enhancement).
- The page is responsive (single column on mobile, multi-column card grid on wider screens).
- PostHog analytics is wired but **only captures from a deployed, non-local production build** -
  local dev and local prod builds never report.
- `npm run build` produces a standalone output, `npm run lint` is clean, and `npm test` passes.
- All copy is ASCII-only and follows `../.github` language rules (voice "you", terse, no hype).

## Scope

**In:**
- Next.js 16 (App Router) + React 19 + TypeScript app scaffold, npm, `output: "standalone"`.
- Tailwind v4 (PostCSS) wired to Canopy: `@rogueoak/roots` tokens + preset, `@rogueoak/canopy`
  components, `@rogueoak/icons`, `@fontsource-variable/figtree` + `@fontsource-variable/geist-mono`.
- The single `/` route with the four sections above, built from Canopy components (`Card`, `Button`,
  `Badge`, `TopNav`/footer primitives) and Canopy semantic-token utilities only.
- Dark-mode toggle + no-flash theme bootstrap script.
- Scroll-reveal animation (fade + rise on viewport entry) via an `IntersectionObserver`-based
  component, honoring `prefers-reduced-motion`, with a no-JS fallback that shows all content.
- PostHog client analytics via a same-origin `/ingest` reverse proxy, gated to production/non-local.
- SEO/OG basics: `metadata`, `sitemap.ts`, `robots.ts`, an OpenGraph image, favicon/manifest.
- Project logo assets copied from `../.github/profile/assets/` into `public/`.
- A `.env.example` documenting every env var by name (no secret values).
- A minimal test (Node built-in runner, `node --test`) proving the analytics gating logic and any
  pure content/data module.
- `README.md` with local dev instructions.

**Out (later specs or later versions):**
- Contact form / Resend email (deferred - not in v1).
- Multi-page content: blog, per-product deep-dive pages, about, resume.
- GitHub Actions CI (0002), Dockerfile (0003), release pipeline (0004), DigitalOcean deploy (0005).
- A bespoke Rogue Oak color theme beyond Canopy defaults (v1 uses Canopy's moss/bark/stone/amber
  semantic tokens as-is; a custom palette override can come later).

## Approach

Mirror matthewmaynes.com's proven structure (it is the reference consumer of Canopy) rather than
inventing a new setup.

**Scaffold & Canopy wiring** (the shared foundation this and later specs reference):
- `package.json`: `next@16`, `react@19`, the three `@rogueoak/*` packages at their published
  versions, the two `@fontsource-variable/*` fonts, `@tailwindcss/postcss`. Scripts `dev`, `build`,
  `start`, `lint`, `test` matching mmc.
- `next.config.ts`: `output: "standalone"`, `outputFileTracingRoot`, the PostHog `/ingest` rewrites
  to US Cloud, and the same security headers (`X-Content-Type-Options`, `X-Frame-Options: DENY`,
  `Referrer-Policy`, `Permissions-Policy`).
- `src/styles/globals.css`: the canonical import chain - `@import 'tailwindcss'` ->
  `@rogueoak/roots/tokens.css` -> `@rogueoak/roots/tailwind-preset.css` -> `@fontsource` fonts ->
  `@custom-variant dark`; plus `@source '../../node_modules/@rogueoak/canopy'` so Tailwind scans the
  component classes.
- **RSC boundary gotcha:** Canopy's published dist has no `"use client"` directives and evaluates
  React context at module scope. Re-export every Canopy component through one `"use client"`
  `src/components/ui.ts` barrel and import from there, exactly as mmc does, to avoid a
  `createContext is not a function` build failure.

**Page & content:**
- One `src/app/page.tsx` composing the four sections as components under `src/components/`.
- Content (product pitches, benefits, the oak story) lives in a small typed data module
  (`src/lib/content.ts` or similar) so copy is reviewable in one place and testable.
- Copy is drawn from the researched, verbatim product pitches and follows the Rogue Oak voice: no
  banned hype words ("revolutionary", "seamless", "powerful", ...), address the reader as "you",
  spaced hyphens not em/en dashes.
- **Copy tone:** confident present tense for all three products - describe what each does today, as
  a shipped tool. No "early development" / "coming soon" / "work in progress" caveats on any product
  (Canopy included); the work is further along than the repos let on.

**Scroll reveal:**
- A small `"use client"` `Reveal` wrapper using `IntersectionObserver` adds an "in view" class that
  drives a CSS fade + upward translate; it unobserves after the first reveal so sections animate
  once. When `prefers-reduced-motion: reduce` is set, or JS is absent, content renders fully visible
  with no transform. Keep it dependency-free (no animation library) to stay light.
- Project links: Spectra/Trellis/Canopy repos under github.com/rogueoak; Canopy Storybook at
  rogueoak.github.io/canopy.

**Analytics:**
- Same split as mmc: a `posthog-provider` mounted once, module-scope init with `api_host: "/ingest"`,
  `capture_pageview: false` + manual pageview component, gated by an `analytics-env` helper that only
  returns true for a production build served from a non-local host.
- The publishable `NEXT_PUBLIC_POSTHOG_KEY` gets a committed default for the Rogue Oak PostHog
  project - `phc_mCAPSQrDTNe6f5KF7kFNgzNvGEFskkKSkw9jru7ej7bW` (publishable keys are safe to commit
  and ship in the browser bundle). No server-side capture in v1 (that rides on the deploy spec if
  wanted later).

**Assets & brand:**
- Copy `rogueoak-logo.svg`, `rogueoak-avatar.svg/png`, `spectra-logo.svg`, `trellis-logo.svg`,
  `canopy-logo.svg` from `../.github/profile/assets/` into `public/`. The new repo is standalone and
  cannot reach a sibling repo at build/Docker time, so the assets must live in-repo.
- Visual identity derives from Canopy tokens (moss/bark/stone/amber, Figtree + Geist Mono) - the
  earthy, tree-themed system - so the site looks like the rest of Rogue Oak by construction.

**Public-repo hygiene:** no secrets in the repo. Only the publishable PostHog key is committed;
`.env.example` lists names only; any private value is documented as belonging in deploy-time
secrets (handled in 0005).
