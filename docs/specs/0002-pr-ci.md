# 0002 - Pull-request CI

## Problem

Changes to rogueoak.com currently merge with no automated checks. A PR can break the lint rules,
the build, or the tests and nobody would know until it is on `main` (and, once deploy exists, until
it fails in production). We need a gate that runs on every pull request so a red check blocks the
merge.

This is also the shared verification step the later deploy pipeline (0005) will reuse, so it is
introduced here as a reusable workflow rather than inline.

Audience: anyone (human or agent) opening a PR against this repo.

## Outcome

- Opening or updating a PR against `main` runs a **verify** job that: installs deps with `npm ci`,
  runs `npm run lint`, `npm run build`, and `npm test`.
- A failing step fails the check; a passing run reports green. The check can be made a required
  status check in branch protection.
- Pushing a new commit to a PR cancels the previous in-flight run for that PR.
- The verification is a **reusable** workflow (`workflow_call`) so the deploy pipeline can run the
  identical gate later - a check can never pass on a PR but fail post-merge.
- Third-party actions are pinned to commit SHAs, and Dependabot keeps those pins current.

## Scope

**In:**
- `.github/workflows/verify.yml` - reusable job (lint, build, test) on Node 24, `cache: npm`.
- `.github/workflows/ci.yml` - `on: pull_request` to `main`, calls `verify.yml`, with
  cancel-in-progress concurrency.
- `.github/dependabot.yml` - weekly `github-actions` updates.
- SHA-pinned actions (`actions/checkout`, `actions/setup-node`) with the version in a trailing
  comment.

**Out (later specs):**
- Building/pushing a container image (0004) and deploying (0005) - `verify.yml` is written to be
  reused there, but this spec only wires the PR gate.
- Branch-protection configuration itself (a GitHub repo setting, done in the UI, not in-repo).
- Caching beyond `actions/setup-node`'s npm cache; matrix builds; E2E/browser tests.

## Approach

Mirror matthewmaynes.com's CI, which already uses a reusable `verify.yml` shared by PR and deploy.
Drop the mmc-specific `resume:pdf:check` step (this repo has no resume). Node 24 matches the
`node:24-alpine` image the container spec (0003) will use, so CI and production run the same major.

- **`verify.yml`**: `on: workflow_call`, `permissions: contents: read`. Steps: checkout ->
  setup-node (24, `cache: npm`) -> `npm ci` -> `npm run lint` -> `npm run build` -> `npm test`.
- **`ci.yml`**: `on: pull_request: branches: [main]`, `permissions: contents: read`,
  `concurrency: { group: ci-${{ github.event.pull_request.number }}, cancel-in-progress: true }`,
  one job that `uses: ./.github/workflows/verify.yml`.
- **Pinning**: reuse the known-good SHA pins from the sibling repo
  (`actions/checkout` v4, `actions/setup-node` v6.4.0) with the tag as a trailing comment, and add
  `dependabot.yml` (github-actions, weekly, commit prefix `ci`) so the pins stay current.

## Acceptance

- [ ] A PR against `main` triggers the `verify` job, which runs `npm ci`, lint, build, and test.
- [ ] The workflows are valid YAML and parse (actionlint / GitHub accepts them; the referenced npm
      scripts all exist in `package.json`).
- [ ] `verify.yml` is reusable via `workflow_call` and `ci.yml` calls it (no duplicated steps).
- [ ] A new push to a PR cancels the prior in-flight CI run.
- [ ] All third-party actions are pinned to a commit SHA with a version comment.
- [ ] `dependabot.yml` is present for the `github-actions` ecosystem.
- [ ] The verify steps pass against the current `main` (lint, build, test all green).
